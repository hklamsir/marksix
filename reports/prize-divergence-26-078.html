核心結論:系統對全部 7 個獎項的中獎注數都與 HKJC 官方公布數值差 10 倍。 此外,二獎 / 四獎 / 六獎 / 七獎 還額外丟失了 0.5 或 0.6 的小數位精度 (官方如「11.5 注」,系統只記成整數「12 注」,再被前端除以 10,顯示為「1.2 注」)。

二、計算鏈路定位

中獎注數從 HKJC API 到前端顯示,共經過兩個處理環節。問題出在兩個環節的運算規則衝突。

HKJC GraphQL 回傳原始資料 → pipline/fetch_prizes.py parse_prizes() → pipline/daily_update.py _parse_prizes() → data/data.js (內嵌) → js/app.js renderPrizes() → 瀏覽器顯示 七獎 範例: lotteryPrizes[7].winningUnit = 3,253,166 → winners = (3253166 + 10//2) // 10 = 325,317 ← 整數除法,小數 .6 被無條件進位 → data.js 儲存: 七獎.winners = 325,317 → (325317 / 10).toFixed(1) = "32531.7" ← 又被除以 unitBet (10) → 瀏覽器顯示: 七獎 32,531.7 注 ← 應為 325,316.6 注

Bug 定位

編號檔案 / 行號現行邏輯問題
Bug A pipline/fetch_prizes.py:119
pipline/daily_update.py:257 		"winners": (wu + ub // 2) // ub if ub else wu 原始 winningUnit 已被 HKJC 設計為「中獎注數 × unitBet」(10 × 注數)。 再除一次會把 11.5 注 等小數值無條件進位成 12 注,破壞官方公布的精度。
Bug B js/app.js:310, 532, 971 (p.winners / unitBet).toFixed(1) p.winners 在 Python 端已被四捨五入成「整數注數」(Bug A 的結果), 前端再除 unitBet = 額外除以 10,顯示值變成實際的 1/10。 註解 // 中獎注數 = prize.winners / unit_bet 本身邏輯正確,但前提應是 winners 為原始 winningUnit。

四、為什麼截圖顯示「325,316.6 注」(與官網一致)?

用戶提供截圖中七獎顯示為 325,316.6 注,這正符合 HKJC 官網。但目前 data/data.js 中 七獎的 winners = 325,317,理論上前端計算結果應為 32,531.7 注。 此矛盾說明截圖是在 更早版本 抓的,當時資料管線應為:

舊版(截圖版本) — 一致正確: winningUnit = 3,253,166 → winners 直接儲存 = 3,253,166 → (3253166 / 10).toFixed(1) = "325316.6" → 顯示 325,316.6 注 ✓ 當前版本 — 出現 Bug: winningUnit = 3,253,166 → winners = (3253166 + 5) // 10 = 325,317 ← Bug A:多除一次 → winners 儲存 = 325,317 → (325317 / 10).toFixed(1) = "32531.7" ← Bug B:又多除一次 → 顯示 32,531.7 注 ✗ (應為 325,316.6 注)
推測:曾有人發現原始 winningUnit 數值太大(例如 3,253,166)看起來「不像注數」, 於是在 fetch_prizes.py 加了 (wu + ub // 2) // ub 想把它轉成「合理的整數注數」, 但 沒同步修改 app.js 的除法。結果從「正確但看起來奇怪」變成「錯 10 倍且毀精度」。

六、修正方案(待您擇一)

方案 A — 一致保留「注數 = 原始 winningUnit / unitBet」(推薦)

思路:Python 端完全不處理注數,直接儲存 HKJC 原始 winningUnit;JS 端維持現有除法。

優點:改動最小、保留 HKJC 原始精度(支持小數位)、跨期(unitBet=5 或 10)通用。

改動:

  • fetch_prizes.py:115-120 — winners 直接存 wu(去掉整除邏輯)
  • daily_update.py:255-258 — 同上
  • app.js — 無需改動,公式 (p.winners / unitBet).toFixed(1) 已正確

風險:重跑 build_data_js.py 或 daily_update.py 後所有期數的注數會更新,前端須重整才生效。

方案 B — Python 端儲存「整數注數」、JS 端移除除法

思路:維持現有 Python 邏輯(四捨五入為整數),但前端不再除以 unitBet。

優點:前端公式更直覺(p.winners 就是注數)。

缺點:會永久丟失小數精度(11.5 → 12,969.5 → 970),與 HKJC 官網公布值無法完全對齊。

改動:

  • Python:不動
  • app.js:310 — 改為 ${p.winners.toLocaleString()}
  • app.js:532 — 改為 ${p.winners.toLocaleString()} 注
  • app.js:971 — 改為 ${(prizeAmount / unitBet).toLocaleString()} 之類直接取 winners

方案 C — Python 改為儲存浮點注數(原始精度)、JS 不動

思路:winners = wu / ub(浮點),JS 端 (p.winners / unitBet).toFixed(1) 改為 p.winners.toFixed(1)。

優點:精確還原 HKJC 官方小數位(11.5、969.5 等)。

缺點:data.js 體積略增(浮點字串);改動兩處。

改動:

  • fetch_prizes.py / daily_update.py — winners = wu / ub(浮點)
  • app.js — 改為 p.winners.toFixed(1)
